3.2. customizing our site

N

ext, we have to modify the example site to adjust it to our own requirements and introduce our own contents instead of the existing one. Freshly unpacked files represent an example site containing these instructions we are currently reading. There are several files and folders of our interest:

  • index.html - this is the main file containing visible project title, social media promotional informations, and meaningful SEO texts enclosed within noscript tags.
  • moonlight.html - this file usually won't need to be referenced unless we want override the default index.html interface. However, if we want to use use our own layout, we may place the moonlight.html reference inside custom iframe structures as a part of user defined index.html.
  • moonlight.json - in this file we can setup various Moonlight rendering parameters such as ovals and background color, power consumption, etc.
  • contents/ - this is a folder where we keep our site content files. contents.xml is the main XML file under the same folder where we structure the site contents. The site content files (HTML, CSS, images, ...) are originally meant to be placed under the contents/ folder and referred from contents.xml file. Nevertheles, the content files may reside in any folders, but it is recommended to place the entire structure under the contents/ folder to keep our site clean and tidy.

Hopefully, examining the default site example (that is, files containing these instructions) should be descriptive enough to compose our own site. This is the quirky part where we will be using file manager and text editor to arrange and edit files of our interest. Please refer to provided links if you need to get more familiar with mentioned notions. This procedure is what makes Fract-exp site creating experience "slack", and requires a bit more advanced knowledge from the site creators, especially considering possible use of server side scripting or other technologies.

Upon each edit, we would want to test if changes we made reflect what we want them to be. Again, we can test them by entering the local HTTP server address into a web browser.
